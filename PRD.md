# Product Requirements Document (PRD): TESS Terminal Pro (Edge Deployment)

## 1. Product Vision
**TESS (Terminal Edge-System Surrogate)** is a locally hosted, privacy-first AI desktop assistant. Unlike cloud-only assistants that suffer from latency and require constant internet connectivity, TESS acts as the "Brain" on your desk. 

**The Goal**: To build a snappy, offline-capable assistant that can parse natural language, control your local desktop applications, manage files, and interact with the physical environment (lights, sensors) in real-time, all while running entirely on low-power edge hardware.

---

## 2. Target Hardware Architecture
To achieve sub-second latency and true edge computing, TESS is decoupled into two physical components:

### 2.1 The Brain (Raspberry Pi 4 / 5)
The primary computational hub responsible for all heavy lifting.
- **Model Execution**: Runs the custom **TESS 152M Parameter SLM (Small Language Model)** entirely in RAM (taking < 500MB).
- **Audio Processing**: Handles Speech-to-Text (STT) parsing of the wake word and user commands, and Text-to-Speech (TTS) for responses.
- **Memory**: Hosts the local Vector Database (RAG) for user identity, calendar events, and preferences.
- **Cloud Router**: A backend service that quietly hands off complex, heavy requests (e.g., "Write a 500-line React application") to cloud LLMs (Gemini/GPT-4) while handling everything else natively.

### 2.2 The Hands & Eyes (ESP32 Microcontroller)
The physical interface responsible for interacting with the real world securely.
- **Communication Protocol**: Connects to the Pi via Serial/USB or local MQTT.
- **Actuators**: Controls relays (e.g., turning on desk lights, fans) based on JSON commands (`{"action": "toggle_pin", "pin": 4}`) generated by the Pi.
- **Sensors**: Reads temperature, humidity, or motion sensors and feeds that context constantly to the Pi.
- **Display**: Drives a small OLED or E-Ink display to show TESS's "face", status, or minor notifications without requiring a full GUI on the Pi.

---

## 3. Software Architecture & The 152M Engine
The heart of TESS is a custom-trained **152-Million Parameter Transformer**. 

**Why not an off-the-shelf 3B model (like Phi-3)?**
Because 3B models melt Raspberry Pis, run at a sluggish 1-token-per-second, and consume all system RAM. TESS 152M is designed for **speed over absolute knowledge**.

* **Architecture**: Modern LLaMA V2.0 style (RoPE embeddings, SwiGLU activations, RMSNorm).
* **Pre-training Data**: Trained on 40,000+ steps of Wikipedia (facts), OpenWebText (slang/culture), TinyStories (grammar), and Hunglish multi-turn conversations.
* **Core Output**: TESS is trained specifically to output structured JSON tool calls instead of just chatting. Example: When the user says *"Open Chrome"*, TESS instantly generates `{"tool": "launch_app", "target": "chrome.exe"}` instead of writing an essay about web browsers.

## 4. Core Capabilities & User Stories

### Feature 1: The Offline Fallback (Identity & Basic Chat)
* **User Story**: As a user, if my Wi-Fi drops, I want TESS to still respond to me so the system doesn't feel "dead."
* **Requirement**: The 152M model natively understands its identity (Name, Creator) and can hold basic, sassy conversations entirely offline with sub-150ms reasoning time. 

### Feature 2: Instant Tool Execution (JSON Generation)
* **User Story**: As a user, I want to say "TESS, create a new React project called 'dashboard' on my Desktop," and have it execute instantly.
* **Requirement**: TESS must parse the intent and generate the strict JSON command (`{"action": "terminal", "command": "npx create-react-app..."}`). The Pi intercepts this JSON and executes it directly on the host machine.

### Feature 3: Physical IoT Control
* **User Story**: As a user, when I say "TESS, it's dark in here," I want the actual desk lamp to turn on.
* **Requirement**: TESS parses the context, generates a `toggle_light` action, and the Pi routes that command over Serial to the physical ESP32 board, which flips the relay. 

### Feature 4: The Smart Cloud Router
* **User Story**: As a user, I want TESS to handle simple tasks instantly, but if I ask it a profoundly difficult coding question, I still want a brilliant answer.
* **Requirement**: The Python backend on the Pi must have a routing mechanism. If the 152M model detects a complex query, it triggers a `{"route_to": "gemini"}` action, effectively acting as an intelligent load balancer.

## 5. Development Roadmap

### Phase 1: The Engine (Currently In Progress)
* [x] Finalize Model Architecture (152M parameters, RoPE/SwiGLU).
* [x] Establish Data Pipeline (Wiki, WebText, Hinglish).
* [x] **In Progress**: Complete Stage 1 Pre-training on Google Colab to teach the model basic grammar, facts, and reasoning.

### Phase 2: The Discipline (Future)
* [ ] Switch to LoRA fine-tuning.
* [ ] Train the model entirely on specific JSON tool-calling formats (e.g., teaching it how to output strict `{"action": "..."}` strings).
* [ ] Inject the "Sassy/Helpful JARVIS" personality traits.

### Phase 3: Hardware Integration (Future)
* [ ] Export PyTorch model to `.onnx` for C++ inference directly on the Raspberry Pi's CPU.
* [ ] Write the Python script on the Pi to handle the Microphone (STT) and Speaker (TTS).
* [ ] Write the C++ code for the ESP32 to listen for Serial commands and toggle relays.
* [ ] Integrate the Pi and ESP32 physically on the desk!
